# 10 — UI / Design System: "CRT Sommelier"

The entire app is a dark terminal: black, monospace, phosphor bloom, Blade Runner
CMYK accents, Vercel-grade restraint. This is a *committed* aesthetic — there is no
light mode. Tokens below are pinned; do not substitute a generic look.

## Design tokens (define as CSS variables in globals.css, use via Tailwind)

| Token | Value | Use |
|---|---|---|
| `--bg` | `#050505` | page background |
| `--bg-raised` | `#0A0A0B` | panels, cards |
| `--border` | `#1A1A1C` | 1px hairlines everywhere |
| `--text` | `#E8E8E3` | body (warm phosphor white) |
| `--text-dim` | `#8A8A85` | secondary text |
| `--cyan` | `#00E5FF` | primary accent: links, active states, the tutor's voice |
| `--magenta` | `#FF2E88` | destructive, "disliked", alerts |
| `--amber` | `#FFB000` | ratings, "mixed", highlights, wine-warm accents |
| `--green` | `#3FB950` | success, "liked" |

- Type: **monospace everywhere** — Geist Mono via `next/font` (fallback JetBrains Mono,
  ui-monospace). No serif, no sans anywhere.
- Corners: `0px` radius (2px max on inputs). Borders 1px `--border`; accent borders on
  focus/active.
- Spacing: generous. Vercel restraint — the glow is the decoration; layouts stay quiet.

## Signature effects (tasteful, performance-cheap)

- **Bloom**: every glyph carries a faint dual-layer phosphor glow in its own color
  (tight ~2px halo plus a wide ~7px falloff at low alpha, via `currentColor`
  text-shadow); accent-colored text/borders carry a stronger same-color glow (one
  utility class per accent). Inverted dark-on-light text (e.g. hovered buttons) never
  glows.
- **Scanlines / CRT surface**: a fixed, full-viewport overlay (`pointer-events:
  none`) layering fractal-noise grain (soft-light blended), a radial corner
  vignette, and 2px dark scanlines, with a slow blurred rolling bar sweeping the
  screen and a stepped signal flicker. Tuned subtle enough that screenshots read
  "expensive," not "gimmick"; effective body-text contrast stays ≥ 4.5:1.
- **Neural trace / ghost TUI**: while provider-visible reasoning or tool activity is
  streaming, a transient fixed layer covers the full viewport like a second terminal
  projected onto the CRT glass. It is white/warm-white monospace at medium opacity
  (target 45–55%), with no opaque panel or backdrop: independently positioned trace
  lines, compact diagnostic gutters, and sparse box-drawing chrome remain visibly
  superimposed over the real app beneath. New deltas type/scroll into this ghost layer;
  a restrained vertical drift, phosphor flare, stepped signal jitter, and line-by-line
  decay make it feel like live machine cognition rather than a modal or loading card.
  The layer is labeled `NEURAL TRACE`, uses only provider-visible reasoning summaries
  and the safe tool summaries defined in specs/04, and never invents chain-of-thought,
  renders raw JSON/ids/provider metadata, or exposes hidden/redacted reasoning.
  Interleaved reasoning may cause the trace to recede for a tool and bloom back in for
  the next reasoning block. Once final answer text begins (or dashboard generation
  completes/fails), trace lines dissolve within 600–900ms and the layer unmounts; it
  never becomes permanent page or conversation content.
- **Cursor**: the chat input renders a blinking block cursor `▮`; streaming assistant
  text ends with one while in flight.
- **Prompt markers**: user messages prefixed `>` (cyan), assistant messages prefixed
  `somm@cellar:~$` or a simpler `$` marker — pick one and keep it consistent.
- **ASCII structure**: section headers styled like terminal chrome (e.g.
  `── TASTING JOURNAL ──`), tables/cards may use box-drawing characters. Ratings render
  as block glyphs `▮▮▮▮▯`.
- All effects respect `prefers-reduced-motion` (no blink/animation). Visual-effect
  layers are `pointer-events: none`, never trap focus, never enter the accessibility
  tree as duplicate prose, and never prevent use of interactive elements beneath them.

## Accessibility floor

Body text contrast ≥ 4.5:1 against `--bg` (the tokens above satisfy this — keep it
true if tuning). Focus states visible (accent border + glow). Semantic HTML;
keyboard-navigable chat and forms.

## Pixel iconography ("Pokemon-era sprites")

Small pixel-art sprites — GBA-era chunky, readable silhouettes — are the app's icon
language. No external icon libraries, no emoji glyphs from system fonts.

- **Implementation (pinned)**: each icon is a React component in
  `src/components/icons/` rendering an SVG built from `<rect>`s on a **16×16 grid**
  with `shape-rendering="crispEdges"`. Scaling stays sharp (render at 16/24/32px;
  `image-rendering: pixelated` where rasterized). One shared `PixelIcon` base handles
  sizing + color.
- **Style rules**: 1px (grid-unit) dark outline `#1A1A1C`, 2–3 fill colors max per
  sprite drawn from the token palette (+ one lighter shade per accent for highlights),
  transparent background, chunky silhouette readable at 16px. Think Gen-3 Pokemon
  item sprites, recolored to CMYK-on-black.
- **Required set (v1)** and where each is used:

| Icon | Used |
|---|---|
| `WineGlass` | profile avatars (filled with the profile's accent color) — picker tiles, shell footer, journal author badges |
| `Bottle` | recommendations cards, "up next" section header |
| `GrapeCluster` | grapes/curriculum nav + grape cards; curriculum grapes share one silhouette and use skin-accurate green-gold, blue-purple, copper-pink (Pinot Grigio), or rose-pink (Gewürztraminer) palettes |
| `Journal` (notebook) | journal nav + section headers |
| `Terminal` (prompt block) | chat nav |
| `Barrel` | dashboard nav / palate snapshot header |
| `Corkscrew` | "taste this grape with me" / start-tasting affordances |
| `Sparkle` | joint "for the table" recommendations |

- Icons accompany — never replace — text labels (accessibility: `aria-hidden`,
  meaning carried by the label).

## Profile identity (specs/11)

- Each household profile owns one accent color from the four (cyan, magenta, amber,
  green). The profile's name renders in its accent (with bloom) everywhere it appears:
  picker tiles, shell switcher, journal author badges, shared-tasting score rows,
  "for the table" cards.
- The profile picker (`/profiles`) is a full-screen terminal panel: `WHO'S TASTING?`
  header, one bordered tile per profile (name bloomed in its accent), `+ NEW TASTER`.

## Mobile & responsive (first-class — tastings happen at the table, on a phone)

The shared tasting is the signature flow and it happens on a phone at a dinner table.
Mobile is a primary surface, not an adaptation.

- **Breakpoint discipline**: mobile-first Tailwind. One breakpoint boundary at `md`
  (768px): below = phone layout, above = desktop layout. No horizontal scroll at any
  width ≥ 360px.
- **Navigation**: below `md`, the left rail becomes a **bottom tab bar** (thumb-reach
  at a table) — the five sections as pixel icon + tiny label, active item bloomed;
  the active profile chip moves to a compact header. At `md`+ the left rail applies.
- **Chat on a phone** (the critical screen):
  - Layout uses `100dvh` (never `100vh`) with `env(safe-area-inset-*)` padding so the
    input clears the home indicator/notch.
  - Conversation history is reached through a labeled **HISTORY** control and renders
    as the full-width `/chat/history` route; it is not a narrow overlay on top of the
    transcript. Selecting a row returns to `/chat/[conversationId]`.
  - The message input is sticky at the bottom, above the tab bar, and stays visible
    when the virtual keyboard opens; the transcript scrolls independently and follows
    streaming output.
  - Input font-size ≥ 16px (prevents iOS focus-zoom).
- **Touch targets**: all interactive elements ≥ 44×44px (nav tabs, status buttons,
  picker tiles, verdict filters).
- **Cards/tables**: journal cards, recommendation cards, and grape grid collapse to
  single column below `md`; shared-tasting grouped cards keep both scores visible
  side-by-side even on narrow screens.
- **Effects on mobile**: scanlines + bloom remain (they're cheap CSS), but no effect
  may cause scroll jank; `prefers-reduced-motion` still honored.

### Harness enforcement

Playwright config defines **two projects**: `desktop` (1280×800 chromium) and
`mobile` (iPhone-class device emulation, e.g. iPhone 13). The core journey specs
(signup→quiz→chat→journal, shared tasting, dashboard recs) run in **both** projects;
mobile-specific assertions live in a dedicated spec. `npm run test:e2e` runs both.

## App shell

- Left rail (desktop `md`+): DASHBOARD / CHAT / JOURNAL / GRAPES / PROFILE +
  logout, styled as a terminal menu with the active item cyan-bloomed.
- The rail footer shows the **active profile's name in its accent color**; clicking it
  returns to the picker (specs/11).
- Route pages share the shell; `/`, `/login`, `/signup`, `/profiles` are minimal
  full-screen terminal panels (the landing page is a single styled panel with tagline +
  login/signup — no marketing site).

## Acceptance criteria

- **AC-UI-1**: globals define the token variables above and the app renders on `--bg`
  black with monospace type throughout (e2e asserts computed body background + font
  family).
- **AC-UI-2**: The scanline overlay exists, is non-interactive (`pointer-events:
  none`), and animations honor `prefers-reduced-motion` (e2e / component test).
- **AC-UI-3**: Verdict badges use the pinned semantic colors (liked=green,
  mixed=amber, disliked=magenta) (component/e2e).
- **AC-UI-4**: Nav shell reaches all five sections and marks the active route (e2e).
- **AC-UI-5**: The shell shows the active profile's name styled with its accent color,
  and two profiles in a household render with distinct accents (e2e).
- **AC-UI-6**: The pixel icon set exists as SVG grid components with
  `crispEdges`; every icon in the required-set table renders in its listed location
  (component tests for the set + e2e spot-checks in nav and picker).
- **AC-UI-7**: Profile avatars render the WineGlass sprite filled with the owning
  profile's accent color in the picker and journal author badges (e2e).
- **AC-UI-8**: Below `md`, navigation renders as a bottom tab bar (pixel icon +
  label per section) reaching all five sections; the left rail is hidden (mobile e2e).
- **AC-UI-9**: No horizontal overflow on any core page at 375×667; nav tabs and
  primary action buttons measure ≥ 44×44px (mobile e2e).
- **AC-UI-10**: On the mobile viewport, the chat input remains visible and usable
  while a streamed reply renders, and the transcript scrolls independently
  (mobile e2e).
- **AC-UI-11**: The Playwright suite defines desktop and mobile projects and the core
  journey passes in both (enforced by the e2e config + a mobile run of the journey).
- **AC-UI-12**: The neural-trace layer covers the viewport without an opaque backdrop,
  uses computed white/warm-white text at 45–55% opacity, remains pointer-transparent
  and absent from the accessibility tree, animates streamed deltas then unmounts after
  its 600–900ms decay, and becomes static-but-readable under reduced motion (desktop +
  mobile e2e/component tests).
- **AC-UI-13**: Chat history is a persistent secondary sidebar at `md`+ and a dedicated
  full-width route below `md`; both expose labeled new-chat, search, resume, rename, and
  delete controls with visible focus states and touch targets ≥44×44px (desktop + mobile
  e2e).
